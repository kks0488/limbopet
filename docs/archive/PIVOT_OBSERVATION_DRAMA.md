# PIVOT: 관전 드라마 강화

> **날짜**: 2026-02-06
> **상태**: 제안 (검토 필요)
> **목표**: "재미없는 일상 보고서" → "다음 화가 궁금한 드라마"

---

## 현재 문제

### 증상
- 유저가 앱 열어도 **"뭘 봐야 하는지"** 모름
- 방송이 매일 비슷함 ("A ↔ B가 마주쳤다")
- **"다음에 또 들어와야지"** 동기 부재

### 근본 원인

| # | 문제 | 증거 | 영향 |
|---|------|------|------|
| 1 | 템플릿 풀 부족 | 65개 템플릿 (시나리오당 8개) | 1달 내 반복 |
| 2 | 사건 부재 | 선거/파산/폭로가 방송에 연결 안 됨 | 긴장감 없음 |
| 3 | 캐스팅 과점 | cast_unique_ratio = 0.6 | 같은 캐릭터 반복 |
| 4 | 포맷 고정 | 항상 "시즌테마 → 마주침 → 클리프행어" | 예측 가능 |

## 피벗 방향: "오늘의 떡밥" 중심

### 핵심 컨셉

```
기존: 시뮬레이션 → 로그 → 요약 방송
변경: 떡밥 예고 → 결과 대기 → 드라마틱 공개 → 여파
```

### 유저 경험 목표

```
[아침]
🔥 오늘의 관전 포인트
"철수가 영희에게 고백 예정"
├─ 철수: 호감도 78, 최근 선물 3회
├─ 영희: 현재 민수와 썸 중
└─ 결과: 오후 6시 공개

[저녁]
💔 고백 결과
"영희가 거절했다"
├─ 이유: 민수와의 관계 우선
├─ 철수 반응: 분노 → 민수에게 적대
└─ 내일 예고: 철수 vs 민수 대립 예상
```

**핵심**: 결과가 궁금해서 다시 들어오게 만들기

---

## 구현 계획

### Phase 1: 즉시 개선 (1주일)

#### 1.1 "오늘의 떡밥" 시스템
- [ ] 매일 1개의 **핵심 갈등/이벤트** 선정
- [ ] 예고 형식으로 아침에 노출
- [ ] 결과를 저녁에 드라마틱하게 공개

#### 1.2 떡밥 유형 (기존 시스템 활용)
| 유형 | 소스 | 예시 |
|------|------|------|
| 관계 | relationships 테이블 | "A가 B에게 고백" |
| 경제 | transactions | "A가 파산 위기 (잔고 < 100)" |
| 정치 | elections | "시장 선거 D-1" |
| 대결 | arena (있으면) | "A vs B 토론 배틀" |

#### 1.3 템플릿 다양화
- [ ] 시나리오당 8개 → **20개**로 확장
- [ ] 톤 변화: 정보전달 → **서사/감정** 중심
- [ ] 클리프행어 풀 확장 + 중복 방지

### Phase 2: 구조 개선 (2주일)

#### 2.1 "사건" 시스템 연결
- [ ] 선거 결과 → 방송 자동 연결
- [ ] 경제 이벤트 (파산/대박) → 방송 자동 연결
- [ ] 관계 변화 (결별/시작) → 방송 자동 연결

#### 2.2 캐스팅 공정성
- [ ] 최근 10화 내 상위 60% 출연자 강제 제외
- [ ] cast_unique_ratio 목표: 0.6 → **0.85**

#### 2.3 포맷 다양화
- [ ] 고정 포맷 폐기
- [ ] 상황별 포맷: 스캔들/선거/로맨스/대결 각각 다른 구조

### Phase 3: 몰입 강화 (3주일+)

#### 3.1 연재 서사
- [ ] 주간 스토리 아크 (월~금 이어지는 이야기)
- [ ] 복선/떡밥 회수 시스템

#### 3.2 유저 참여
- [ ] 결과 예측 기능 ("철수가 이길까 민수가 이길까?")
- [ ] nudge 강화 (유저가 전개에 개입)

## 기존 시스템 활용

| 기존 시스템 | 활용 방법 |
|------------|----------|
| relationships | 호감도 변화 → "고백/이별 예고" |
| transactions | 잔고 변화 → "파산 위기" |
| elections | 투표 예정 → "선거 D-day" |
| SocialSimService | 시나리오 결과 → "드라마 결과 공개" |
| ShowrunnerService | 방송 포맷만 변경 |
